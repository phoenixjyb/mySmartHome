# 全屋智能系统总体架构（V3）

**项目：** 约 100 ㎡，三卧室、一客厅、一厨房、两卫生间、两个阳台  
**网络：** 1000M 光纤宽带  
**核心原则：** 有线骨干、无线终端；本地优先；原厂控制保留；重要功能断网可用；系统由业主自行管理  
**本版日期：** 2026-06-24  
**版本：** V3.1（补充格恩中央加湿 Modbus 协议）

## 0. 本版确认的空间用途

- **Bedroom1：办公室/书房**，放置 RTX 3090 工作站，保留 2 个有线网口。
- **Bedroom2：可能放置另一台台式机**，保留 2 个有线网口。
- **Bedroom3：不设置固定网口**，智能传感器使用 Zigbee。
- **Balcony1：Bambu P2S 3D 打印区**，采用简化封闭、防晒、短距离外排方案，保留 1 个备用网口。
- **Balcony2：保留 1 个备用网口**；如放洗衣机，增加水浸传感器。
- **Storage room：网络及自动化设备间**，只放网络机柜、Mac mini、网关和 UPS；不再放 3090 工作站或 P2S。
- **Living room：无线投影仪**，但装修时仍预埋电源、Cat6 和影音空管。

# 2. 全屋传感器架构

## 2.1 传感器分级

### A. 独立安全层

这些设备必须在 Home Assistant、网络或电源服务器故障时仍能独立报警或保护：

- 烟雾报警器；
- 燃气报警器；
- 一氧化碳报警器；
- 配电保护和漏电保护；
- 燃气系统原厂安全保护；
- 电动总水阀的手动操作；
- 空调、加湿器原厂控制器。

### B. 本地自动化层

- Zigbee 温湿度、门磁、水浸、人体/存在传感器；
- CO₂ 传感器；
- Modbus 空调及加湿状态；
- 用电和用水计量；
- 智能锁状态；
- 排风和灯光执行器。

### C. 厂家附加层

- 冰箱、洗衣机、扫地机、油烟机、P2S 等厂商 App 或云接口；
- 只用于状态、提醒和非关键场景，不作为安全和基础控制依赖。

## 2.3 温湿度传感器布置原则

- 普通房间安装在内墙、离地约 1.2–1.5m 的活动区域；
- 避开阳光直射、空调送风口、窗边、电脑和电视散热口；
- Bedroom1 的传感器不要装在 RTX 3090 机箱排风方向；
- 卫生间传感器安装在干区，不直接被淋浴水冲刷；
- Balcony1 传感器放在遮阳、空气流通的位置，不能紧贴打印机排风口；
- 储物间传感器放在机柜进风侧，不装在交换机或 UPS 正上方；
- 厨房温湿度不纳入全屋空调和加湿控制平均值；
- 卫生间、厨房、Balcony1 的湿度不参与中央加湿器的主控湿度计算。

## 2.4 推荐数量

首期建议：

- 温湿度：8 个；
- CO₂：2 个；
- 人体/存在：7–8 个；
- 门磁/窗磁：至少 3 个，卧室可开启窗户按需增加；
- 水浸：5–7 个；
- 独立烟雾报警：客厅/走廊、储物间、Balcony1 至少各 1 个；
- 燃气及一氧化碳报警：按灶具、燃气热水器位置和当地规范配置。

# 3. 网络架构

## 3.1 物理网络

```text
运营商光纤
   │
光猫 / ONT
   │
独立路由器 / 防火墙
   │ 2.5GbE
16 口可管理 PoE+ 交换机
   ├── AP1：左侧走廊
   ├── AP2：客厅
   ├── Mac mini / Home Assistant
   ├── Bedroom1 RTX 3090 工作站
   ├── Bedroom2 台式机
   ├── 中央走廊 PoE Zigbee 协调器
   ├── RS-485 转以太网网关
   ├── 客厅影音柜
   ├── 投影仪备用有线口
   ├── 入户 PoE 摄像头/门铃（可选）
   ├── Balcony1 备用口
   ├── Balcony2 备用口
   └── NAS 未来预留
```

## 3.2 AP 和 Zigbee 协调器位置

- **AP1：** 卧室 1、卧室 2、厨房之间的走廊吊顶；
- **AP2：** 客厅中部偏右吊顶，覆盖客厅、Bedroom3 和两个阳台；
- **AP3：** 只保留空管或备用线，不初期采购；
- **Zigbee 协调器：** 中央走廊开放位置，通过 Ethernet/PoE 接入；
- Zigbee 协调器不放进金属机柜，不与 AP 紧贴安装；
- 2.4GHz Wi-Fi 与 Zigbee 信道在安装调试阶段统一规划，不提前把信道写死。

## 3.3 网线点位

| 位置 | 数量 | 建议 |
|---|---:|---|
| AP1、AP2 | 2 | Cat6，PoE |
| AP3 备用 | 1 | Cat6 或空管 |
| Bedroom1 办公桌 | 2 | Cat6A，工作站和备用 |
| Bedroom2 书桌 | 2 | Cat6A，台式机和备用 |
| Bedroom3 | 0 | 使用 Wi-Fi/Zigbee |
| 客厅影音柜 | 2 | 播放器、功放或本地小交换机 |
| 投影位置 | 1 | Cat6，另加影音空管 |
| Balcony1 | 1 | Cat6A 备用；P2S 本体使用 Wi-Fi |
| Balcony2 | 1 | Cat6 备用 |
| Zigbee 协调器 | 1 | Cat6，PoE |
| 入户门口 | 1 | PoE 摄像头或可视门铃二选一 |
| RS-485 网关 | 1 | 机柜内以太网接入 |
| NAS | 1 | 未来预留 |

所有线路星形回到储物间配线架，中途不得接头。

## 3.4 交换机选择

推荐：

- 16 个 RJ45 端口；
- 至少 8 个 PoE+ 端口；
- PoE 总功率约 100W 或以上；
- 至少 4 个 2.5GbE 端口；
- 最好有 2 个 10G/SFP+ 端口；
- 支持 VLAN、端口隔离、PoE 管理和链路聚合；
- 使用 24 口被动配线架保留扩展余量。

预计首期实际占用约 10–12 个交换机端口，因此 16 口足够。未使用的墙面端口只留在配线架，不需要全部跳接到交换机。

## 3.5 VLAN 和 SSID

| VLAN | 名称 | 设备 |
|---:|---|---|
| 10 | Trusted | 手机、电脑、3090、Bedroom2 台式机 |
| 20 | Automation | Home Assistant、MQTT、Zigbee 协调器、RS-485 网关、ESPHome |
| 30 | IoT | 投影、电视、冰箱、洗衣机、扫地机、P2S 等 |
| 40 | Security | 门口摄像头/门铃、未来 NVR |
| 50 | Guest | 访客设备 |
| 99 | Management | 路由器、交换机、AP 管理 |

建议 SSID：

- `Home` → Trusted；
- `Home-IoT` → IoT；
- `Guest` → Guest。

基本规则：

- IoT 不得主动访问 Trusted；
- Trusted 可访问 Home Assistant 和必要的 IoT；
- Automation 仅允许必要的互联网、DNS、时间和更新；
- Security 仅访问 NVR、DNS、时间和管理端；
- Management 只允许管理员设备访问；
- 远程访问通过 WireGuard，不对公网直接开放 Home Assistant、摄像头或路由器后台。

# 4. RS-485 与暖通架构

## 4.1 大金空调：DTA120A611 候选方案

当前将 **DTA120A611** 作为大金空调接入的候选原厂转换组件，但在取得正式资料前，不把其输出接口和寄存器能力写死。

```text
大金空调系统
      │ 原厂内部通信
      ▼
DTA120A611（候选）
      │
      ├── 情况 A：输出 Modbus RTU / RS-485
      │             ↓
      │      隔离型 RS-485 转 Ethernet 网关
      │             ↓
      │        Automation VLAN
      │             ↓
      │       Home Assistant Modbus
      │
      └── 情况 B：直接输出 Modbus TCP / Ethernet
                    ↓
               Automation VLAN
                    ↓
              Home Assistant Modbus
```

施工和采购前必须向大金供应商取得：

1. DTA120A611 正式安装手册；
2. 与最终室外机、室内机和线控器型号的兼容表；
3. 输出方式：Modbus RTU、Modbus TCP 或其他；
4. 波特率、数据位、校验、停止位和地址规则；
5. 完整寄存器表；
6. 每个房间可读写的功能；
7. 支持的室内机数量；
8. 是否可以读取实际室温、故障码和运行状态；
9. 是否可以写入开关、模式、设定温度和风速；
10. 原厂线控器能否同时保留；
11. 第三方接入是否影响保修。

大金内部总线接线由大金安装方完成，装修电工只负责管线、电源和机柜侧接口预留。

## 4.2 格恩中央加湿：已取得 BG 系列 Modbus 协议

上传的《格恩通信协议列表格式-液晶控制器》属于 **格恩 BG 系列中央加湿控制器**，不是大金空调网关协议。根据表格，当前可确认：

- 协议：Modbus RTU；
- 物理层：RS-485；
- 波特率：9600；
- 串口格式：8N1（8 数据位、无校验、1 停止位）；
- 读取功能码：03H（Holding Registers）；
- 寄存器字长：2 字节，按 16 位无符号整数处理作为首期假设；
- 默认从机地址：表中初始值为 1，地址寄存器说明可设 2–255；
- 建议通过独立的隔离型 RS-485 转 Ethernet 网关接入 Home Assistant。

```text
格恩 BG 系列液晶控制器
        │ RS-485 / Modbus RTU，9600 8N1
        ▼
隔离型 RS-485 转 Ethernet 网关独立端口
        │ Modbus TCP 透传/转换
        ▼
Automation VLAN
        │
Home Assistant Modbus
```

### 4.2.1 已知寄存器映射

| 寄存器 | 权限 | 建议实体 | 已知值/含义 |
|---|---|---|---|
| 0x0001 | R/W | 从机地址 | 初始值 1；表中说明设定范围 2–255 |
| 0x0002 | R/W | 加湿器开关 | `0x01` 开机；`0x10` 关机，注意关机不是 0 |
| 0x0004 | R/W | 手动/定时排水命令 | 0 不排水；1 排水，需确认是否自动复位 |
| 0x0005 | R/W | 目标湿度 | 初始值 60；允许范围和步进待厂家确认 |
| 0x0006 | R | 当前室内湿度 | 整数百分比作为首期假设；量程/缩放待确认 |
| 0x0008 | R/W | 送风风速 | AC 风机 1–3；EC 风机 1–5；表中初始值与备注默认值存在差异，需确认 |
| 0x0009 | R/W | 风机模式 | 0 手动；1 自动；2 睡眠；3 急速 |
| 0x000A | R/W | 定时排水间隔 | 1–99；单位为小时或天，由主板设置决定 |
| 0x0013 | R | 水位状态 | 1 缺水；2 低；3 中；4 高；0x0F 探头故障 |
| 0x0014 | R | 运行状态字 | D7 EC 风机；D6 电磁输出；D5 UV；D4 风机；D2 进水阀；D1 水泵；D0 湿轮电机 |
| 0x0015 | R | 故障字 | D4 湿度传感器；D3 高水位；D2 水位探头；D1 加水；D0 排水 |

### 4.2.2 Home Assistant 中建议建立的实体

- `switch`：加湿器开/关；
- `number`：目标湿度；
- `sensor`：当前湿度、水位、状态字、故障字；
- `select`：风速、风机模式；
- `button` 或受保护脚本：手动排水；
- `binary_sensor`：把状态字和故障字的各个位拆分为运行/故障状态；
- 告警自动化：缺水、高水位、水位探头、加水、排水和湿度传感器故障。

### 4.2.3 接入前仍须厂家书面确认

1. R/W 寄存器使用 06H 还是 10H 写入，或两者均支持；
2. 寄存器地址在网关/Home Assistant 中采用 0 基还是 1 基；协议写“可从 0x0000 开始读”，但首个已定义寄存器为 0x0001；
3. 目标湿度和当前湿度的有效范围、单位、缩放及异常值；
4. 0x0004 排水命令写入 1 后是否自动复位，以及允许触发条件；
5. 0x0008 风速的初始值 3 与备注“默认 2 档”不一致，最终以哪一项为准；
6. 0x0006 标记为只读，但备注提到“MODBUS 总线回传优先”，是否另有外部湿度写入寄存器；
7. 故障字位定义与 E1–E5 故障代码之间的准确对应关系；
8. 设备地址默认值与可设范围的矛盾：初始值 1，但说明写 2–255；
9. A/B/GND 端子定义、屏蔽接地、终端电阻和最大总线长度；
10. 原液晶控制器与 Modbus 主站能否长期同时工作。

### 4.2.4 串口网关分配

建议采用 **2 路或 4 路相互隔离的工业串口服务器**：

- Port 1：大金空调（仅在 DTA120A611 最终确认输出 RS-485 时）；
- Port 2：格恩中央加湿，固定 9600 8N1；
- Port 3：未来 Modbus 电表或水表；
- Port 4：备用。

空调与加湿器不共用同一条 RS-485 总线。每一路独立设置波特率、校验、超时和设备地址。加湿器建议以 5–10 秒读取湿度和运行状态、30–60 秒读取故障及水位作为调试起点，不频繁写入目标湿度。

## 4.3 控制原则

- 原厂空调线控器继续负责基本控制和保护；
- 格恩自身控制器继续负责基础恒湿和故障保护；
- Home Assistant 只做设定值调整、日程、场景和跨系统联动；
- 不通过智能插座切断中央空调、加湿器或燃气热水器电源；
- 自动化必须设置滞回、最短保持时间和频率限制，避免频繁改变设定值。

# 6. 系统交互架构

## 6.1 数据流

```text
Zigbee 传感器
      │
PoE Zigbee 协调器
      │
Zigbee2MQTT
      │
Mosquitto MQTT
      │
Home Assistant
      │
      ├── Zigbee 灯光、窗帘、排风
      ├── Modbus 大金空调
      ├── Modbus 格恩加湿
      ├── LAN/Wi-Fi 投影和音响
      ├── Wi-Fi 家电和扫地机
      └── 自编应用、通知和仪表盘
```

## 6.2 典型场景

### 办公室舒适场景

```text
Bedroom1 检测到有人
+ 窗户关闭
+ 温度偏高
→ 通过 Modbus 调整大金设定温度
→ 不直接切断空调电源

CO₂ 持续偏高
→ 提醒开窗或提高新风
```

### 中央加湿场景

```text
白天：客厅 + Bedroom1 湿度作为主要参考
夜间：实际有人卧室湿度作为主要参考
卫生间、厨房、Balcony1 不参与全屋平均

门窗打开
→ 暂停或降低加湿设定

湿度过高或存在结露风险
→ 降低目标湿度
```

### 卫生间排风

```text
检测到有人
→ 开灯

湿度快速上升
→ 开排风

湿度下降并接近基线
→ 延时关闭排风
```

### 观影场景

```text
按下 Zigbee 场景按键
→ 关闭客厅窗帘
→ 调暗灯光
→ 开启投影
→ 开启功放/播放器
→ 选择对应输入源
```

### Balcony1 打印场景

```text
检测到 P2S 用电或打印状态
→ 开启局部排风

打印结束
→ 延时 15–30 分钟关闭排风

阳台温度过高
→ 提醒并提高排风

烟雾报警
→ 独立声响报警
→ 手机通知
→ 按经电气人员确认的安全逻辑处理电源
```

### 漏水保护

```text
任一关键水浸传感器报警
→ 本地声光/手机通知
→ 关闭电动总水阀
→ 记录报警位置和时间
```

### 离家场景

```text
入户门关闭/上锁
+ 全屋无人在设定时间内
→ 关闭非必要灯光
→ 空调转节能设定
→ 加湿器降低目标
→ 关闭窗帘（按用户偏好）
→ 启动扫地机器人（条件满足时）
```

## 6.3 故障降级

| 故障 | 应保持可用的功能 |
|---|---|
| 互联网中断 | 本地灯光、窗帘、空调、加湿、传感器和场景继续运行 |
| Home Assistant 停机 | 墙面开关、窗帘按键、空调原厂线控、加湿原厂控制继续使用 |
| Zigbee 协调器故障 | 原厂空调/加湿、普通家电和手动开关继续使用 |
| Mac mini 重启 | 网络路由不受影响；路由器为独立设备 |
| 交换机故障 | 安全报警仍应独立；重要设备可手动操作 |
| 厂商云服务故障 | 核心灯光、窗帘、暖通和漏水保护不依赖该云服务 |

# 7. 软件架构

## 7.1 Mac mini 推荐部署

```text
Mac mini / macOS
├── Home Assistant OS 虚拟机
│   ├── Home Assistant Core
│   ├── Mosquitto MQTT
│   ├── Zigbee2MQTT
│   ├── ESPHome
│   └── Node-RED（可选）
│
└── 独立 Debian/Linux 虚拟机（可选）
    ├── Python / Go / Node.js 自编服务
    ├── Git 仓库
    ├── InfluxDB / Grafana（后期）
    └── 其他非关键应用
```

最简首期方案：全部核心组件运行在 Home Assistant OS 虚拟机及其 Add-on 中。  
当自编应用增多后，再增加独立 Linux 虚拟机，避免开发程序影响核心家庭控制。

## 7.2 软件组件职责

| 组件 | 职责 |
|---|---|
| Home Assistant | 统一设备模型、仪表盘、自动化和权限 |
| Zigbee2MQTT | Zigbee 设备管理、状态和命令转换 |
| Mosquitto | Zigbee2MQTT 与 Home Assistant/自编应用之间的消息总线 |
| ESPHome | CO₂、空气质量、排风、脉冲水表和自制传感器 |
| Home Assistant Modbus | 读取/写入大金、格恩及未来 Modbus 设备 |
| Node-RED | 只用于确有需要的复杂流程，首期非必需 |
| Recorder | 首期保存历史数据 |
| InfluxDB/Grafana | 后期做长期趋势和能耗分析 |
| WireGuard | 安全远程访问 |
| Git/备份 | 保存配置、自动化和自编程序 |

## 7.3 自编应用接入原则

- 读取状态优先通过 Home Assistant WebSocket/REST API；
- 事件和松耦合控制可使用 MQTT；
- 不建议多个应用同时直接轮询同一条 Modbus 总线；
- 由 Home Assistant 或单一 Modbus 服务统一管理寄存器读写；
- 重要控制命令增加权限、频率限制、超时和状态反馈；
- 自动化配置、脚本和寄存器映射纳入 Git 版本管理。

## 7.4 备份和运维

- 每日自动备份 Home Assistant；
- 每周把加密备份复制到另一台设备或云存储；
- Zigbee2MQTT 数据目录和协调器配置单独备份；
- 路由器、交换机和 AP 配置导出保存；
- 对网关、AP、协调器和服务器设置 DHCP 固定租约；
- 建立设备清单：型号、MAC、IP、安装位置、协议、保修及电池型号；
- 建立低电量、离线和高温告警；
- 每半年测试一次漏水关阀、烟雾报警、UPS 和故障降级。
