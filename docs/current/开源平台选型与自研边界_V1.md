# 开源平台选型与自研边界（V1）

**项目：** mySmartHome / smartHouse  
**日期：** 2026-06-26  
**阶段：** 方案设计、硬件选型和软件架构边界确认

## 1. 结论

本项目不从零重做智能家居平台。核心策略是：

```text
采用成熟开源平台作为基础
→ 用配置和标准集成接入大部分设备
→ 对暖通、加湿、运行策略和验收流程编写自研层
→ 安全和手动控制保留原厂路径
```

推荐默认栈：

```text
mySmartHome / smartHouse
├── Home Assistant：本地控制平台、设备模型、仪表盘、自动化入口
├── Mosquitto MQTT：本地消息总线
├── Zigbee2MQTT：Zigbee 设备桥接
├── ESPHome：自制传感器、排风、表计和 ESP32 设备
├── Home Assistant Modbus：只读或低风险 Modbus 实体
├── SmartHouse Core：本项目自研服务，处理受保护写控制和策略逻辑
├── InfluxDB / Grafana：后期长期历史和分析
├── Node-RED：可选，只用于确实更清晰的流程编排
└── Frigate：可选，后期如需要本地摄像头/NVR/目标检测再引入
```

Home Assistant 是现有开源平台，不是本项目名称。本项目名称仍使用 `mySmartHome` / `smartHouse`，自研软件层暂称 `SmartHouse Core`。

## 2. 平台角色划分

| 层级 | 采用项目 | 角色 | 本项目工作方式 |
|---|---|---|---|
| 主控制平台 | Home Assistant | 设备注册、实体模型、仪表盘、自动化、手机端、备份 | 配置、集成和少量扩展 |
| 消息总线 | Mosquitto MQTT | Zigbee2MQTT、ESPHome、自研服务之间的事件和命令通道 | 配置 Topic、权限和备份 |
| Zigbee | Zigbee2MQTT | 温湿度、门磁、水浸、人体/存在、场景按钮、窗帘等 Zigbee 设备 | 设备入网、命名、分组、备份 |
| 自制硬件 | ESPHome | CO2、VOC/PM、机柜温湿度、排风、脉冲表计等 | YAML 配置和必要的硬件封装 |
| 标准 Modbus | Home Assistant Modbus | 日立、格恩、电表/水表的只读状态和简单实体 | 先只读，写控制不直接开放 |
| 受保护控制 | SmartHouse Core | 暖通/加湿写控制、频率限制、回读确认、日志、策略 | 自研 |
| 可视化历史 | InfluxDB/Grafana | 长期温湿度、CO2、能耗、用水趋势 | 后期引入 |
| 复杂流程 | Node-RED | 只有在可视化流程明显更清晰时使用 | 可选，不作为首期必需 |
| 摄像头/NVR | Frigate | 本地录像、对象检测、门口/安全事件 | 后期按实际摄像头需求引入 |

## 3. 不重新发明的部分

不自研以下基础能力：

- 通用设备注册和实体模型；
- 通用智能家居仪表盘；
- 手机 App 和远程通知基础能力；
- MQTT Broker；
- Zigbee 协议栈；
- ESP32 常见传感器固件框架；
- 通用自动化引擎；
- 通用备份、用户权限和集成市场。

这些能力优先使用 Home Assistant、Mosquitto、Zigbee2MQTT、ESPHome 等成熟开源工具。

## 4. 必须自研或强约束的部分

以下内容属于本项目核心价值和风险边界：

1. **日立 Modbus 发现与映射**
   - 读取 `4998`、`4997`；
   - 按 `Y / 系统号 / 地址号 / 房间` 建立映射；
   - 与原厂线控器逐房间对照。

2. **受保护暖通写控制**
   - `4998 != 1` 时拒绝写入；
   - 写队列和频率限制；
   - 同一室内机连续写入至少间隔约 500ms；
   - 写后回读；
   - 所有写入记录发起人、时间、寄存器和值；
   - `4999` 和 `4004` 保持维护级二次确认动作。

3. **格恩中央加湿策略**
   - 独立 RS-485 通道；
   - 保留原液晶控制器；
   - Home Assistant 只做设定值、场景、告警和联动；
   - 缺水、故障、基础恒湿仍由原控制器承担。

4. **跨房间舒适策略**
   - 房间温湿度、CO2、存在、门窗状态；
   - 日夜模式；
   - 卫生间、厨房、Balcony1 不参与全屋湿度主控平均；
   - 防止设备频繁启停。

5. **施工和验收证据**
   - 网线编号和测试结果；
   - 隐蔽工程照片；
   - RS-485 A/B/GND、屏蔽、终端电阻记录；
   - Home Assistant 停机、断网、断电、UPS、原厂控制器回退测试。

## 5. Home Assistant 使用方式

首期采用 Home Assistant OS 或等价可维护部署方式。优先使用官方/成熟集成：

- MQTT；
- Modbus；
- ESPHome；
- Zigbee2MQTT 接入；
- Recorder；
- Backup；
- WireGuard 或等价 VPN；
- 后期按需加入 InfluxDB/Grafana。

本项目不建议直接 fork Home Assistant Core。需要扩展时优先顺序：

1. Home Assistant 标准配置；
2. MQTT/REST/WebSocket 与自研服务通信；
3. 自定义集成；
4. 独立 Add-on/App；
5. 只有确有长期维护理由时才考虑 fork。

## 6. Submodule 策略

Git submodule 可以把另一个 Git 仓库固定到本仓库的某个路径和提交。它适合“需要精确锁定外部代码版本”的情况，但会增加初始化、更新和协作复杂度。

本项目当前不把 Home Assistant、Zigbee2MQTT、ESPHome、Mosquitto 等大型上游项目作为 submodule。原因：

- 它们应通过 Home Assistant Apps/Add-ons、容器镜像、包管理器或官方安装方式部署；
- 我们不计划直接修改这些上游源码；
- submodule 会把本仓库变成上游源码镜像，增加维护噪声；
- 上游项目更新频繁，直接 pin 源码不等于可运行部署版本。

当前建议：

| 场景 | 是否用 submodule | 推荐做法 |
|---|---|---|
| Home Assistant Core | 否 | 记录版本和安装方式；使用官方 HAOS / Add-on / Integration |
| Zigbee2MQTT | 否 | 使用 Add-on、容器或独立服务；配置纳入本仓库 |
| ESPHome | 否 | 配置纳入本仓库；编译/刷写走 ESPHome 工具 |
| Mosquitto | 否 | 记录配置和备份；不 vendoring 源码 |
| 本项目拆出的可复用库 | 以后可考虑 | 先放在本仓库，稳定后再拆分 |
| 必须本地修改的第三方驱动 | 可考虑 | fork 后作为 submodule 或包依赖，并记录 license |
| 厂商 PDF/协议表 | 否 | 作为 `references/` 文件跟踪，不用 submodule |

如果将来添加 submodule，统一放在 `external/` 下，并在 `external/README.md` 记录：

- 上游仓库；
- 锁定 commit；
- 使用原因；
- 更新流程；
- license；
- 与本项目代码的接口。

## 7. License 和合规边界

- Home Assistant Core 为 Apache-2.0 开源项目；
- Zigbee2MQTT 主项目使用 GPL-3.0；
- ESPHome license 同时涉及运行时代码 GPLv3 和 Python/其他部分 MIT；
- 本项目自研代码应单独声明 license，避免无意识复制 GPL 代码进闭源或不兼容模块；
- 配置文件、协议映射、现场验收记录和自研业务逻辑保持在本仓库；
- 对第三方源码只链接、调用或作为独立进程使用，除非明确决定 fork/修改。

建议按以下顺序落地：

1. `docs/current/实施路线图与仓库结构_V1.md`：明确阶段、目录和交付物；

8. SmartHouse Core 的最小可运行服务。
