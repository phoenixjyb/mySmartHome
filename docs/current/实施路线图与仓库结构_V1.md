# 实施路线图与仓库结构（V1）

**项目：** mySmartHome / smartHouse  
**日期：** 2026-06-26  
**目标：** 从装修设计资料包演进为自建智能家居系统的长期工程仓库。

## 1. 总体目标

本仓库同时承载三类内容：

1. **设计和施工基线**
   - 网络、弱电、暖通、传感器、排风、机柜、验收要求；
   - 给设计师、电工、暖通安装方和业主自己使用。

2. **集成和调试证据**
   - 厂商协议、点表、串口参数、网线测试、房间映射、照片索引；
   - 证明“可接入”和“已验证”。

3. **自研软件和运维**
   - Home Assistant 配置；
   - SmartHouse Core；
   - Modbus 调试脚本；
   - 自动化策略；
   - 备份、恢复和验收脚本。

## 2. 阶段划分

### 阶段 0：当前资料基线

已完成：

- V3.2 总体架构；
- V2.1 施工交底；
- 日立 Modbus 点表分析；
- 格恩中央加湿协议表纳入参考；
- Home Assistant 日立只读 YAML 示例；
- GitHub 仓库 `phoenixjyb/mySmartHome`。

### 阶段 1：设计冻结前

目标：把隐蔽工程和硬件采购风险降到最低。

交付物：

- 设备采购候选表；
- 网络端口和配线架编号方案；
- AP、Zigbee 协调器、RS-485 网关、机柜、UPS 位置确认；
- 日立和格恩供应商问题清单；
- 灯光、窗帘、门口视频、总水阀选型边界；
- 衣帽间 P2S 台面、独立插座、可选通风/外排和安全监测确认记录。

不做：

- 不开放暖通写控制；
- 不把 Home Assistant 作为安全系统唯一依赖；
- 不采购只能云端控制且无本地协议的关键设备。

### 阶段 2：装修和布线

目标：把后期不可逆成本最高的内容做对。

交付物：

- 网线和弱电管线编号；
- 所有线缆星形回储物间；
- AP、Zigbee、RS-485、投影影音管、阳台备用口预留；入户高低位本轮不预留；
- 日立和格恩独立 RS-485 通道；
- 开关盒零线和加深底盒；
- 窗帘盒 220V 与弱电空管；
- 软水、净水、洗衣机、热水器、加湿设备水浸点；
- 封墙/吊顶前照片和测试记录。

### 阶段 3：核心平台上线

目标：先建立可观测、可备份、可恢复的本地平台。

交付物：

- 路由器、防火墙、VLAN、SSID；
- 16 口 PoE+ 交换机和固定 DHCP 租约；
- Apple Silicon Mac mini 上的 Home Assistant OS 虚拟机；
- Mosquitto、Zigbee2MQTT、ESPHome；
- PoE Zigbee 协调器；
- 基础传感器入网；
- 每日 HA 备份和每周离机备份；
- WireGuard/VPN 远程访问。

### 阶段 4：只读设备集成

目标：所有关键系统先“看得见”，再谈控制。

交付物：

- 日立 `4998/4997` 只读验证；
- 日立 Y/系统号/地址号/房间映射；
- 格恩湿度、水位、运行状态、故障状态只读验证；
- UPS、机柜温湿度、水浸、门磁、人体/存在、CO2 入库；
- Home Assistant 仪表盘只显示状态，不做危险控制。

### 阶段 5：受保护写控制

目标：在原厂控制器可回退的前提下逐步开放有限写入。

交付物：

- SmartHouse Core 最小服务；
- 日立写队列、频率限制、`4998` 检查和写后回读；
- 格恩目标湿度、开关、风速和排水命令的安全封装；
- 所有写入审计日志；
- 断网、HA 停机、网关故障、原厂控制器回退测试。

### 阶段 6：自动化和体验优化

目标：让系统稳定地提高居住体验，而不是制造复杂性。

交付物：

- 办公室舒适策略；
- 夜间卧室舒适策略；
- 中央加湿策略；
- 卫生间排风策略；
- P2S 打印排风策略；
- 漏水关阀和通知；
- 观影场景；
- 离家/回家场景；
- 长期趋势和维护提醒。

## 4. 软件目录边界

### `software/smarthouse-core/`

未来自研控制服务。职责：

- 统一管理危险写控制；
- 暖通/加湿写队列；
- 写前条件检查；
- 写后回读；
- 审计日志；
- 对 Home Assistant 暴露 REST/MQTT/WebSocket 接口；
- 不直接替代原厂控制器的保护逻辑。

首期可以先写成 Python 服务，后续按需要再评估 Go/Node.js。

### `software/modbus-tools/`

现场调试工具。职责：

- 读取 4998/4997；
- 扫描室内机地址；
- 导出房间映射；
- 验证地址是否需要 `-1`；
- 读取格恩状态字和故障字；
- 生成验收 CSV/JSON。

### `config/home-assistant/`

Home Assistant 配置示例和可审查配置。规则：

- 不提交真实 `secrets.yaml`；
- 不直接提交未审查的现场生成配置；
- 日立首期只读；
- 任何写控制都必须通过 SmartHouse Core 或明确的受保护脚本。

## 5. Submodule 使用规则

默认不使用 submodule。优先使用：

- 官方安装方式；
- Add-on/App；
- 容器镜像；
- 包管理器；
- 配置文件；
- 文档链接和版本记录。

只有满足以下条件才考虑 submodule：

1. 必须使用某个外部仓库的源码；
2. 需要精确锁定 commit；
3. 我们需要本地 patch 或 fork；
4. 该外部代码不会被包管理器更好地管理；
5. `external/README.md` 已记录原因、license 和更新流程。

## 6. 当前优先级

近期最有价值的工作：

1. 建立设备采购候选清单；
2. 建立布线/端口编号表；
3. 建立供应商问题清单；
4. 固定控制主机实施方案：Apple Silicon Mac mini + Home Assistant OS 虚拟机 + 外部备份盘/NAS 备份；
5. 确认路由器、防火墙、交换机、AP、Zigbee 协调器和 RS-485 网关候选；
6. 准备 Home Assistant 首期目录结构；
7. 准备只读 Modbus 现场验证脚本；
8. 再开始 SmartHouse Core 的最小服务。

## 7. Git 工作方式

- 设计决策、施工交底、采购候选和验收模板都进入 Git；
- 私密信息不进入 Git；
- 每个阶段用小提交记录；
- 大的方案变化先写设计文档，再改配置或代码；
- 硬件触发的配置变化必须带现场证据或供应商资料引用。
